# WindowsBlueToothManager 系统设计与实现文档

## 1. 项目概述

WindowsBlueToothManager 是一个面向 Windows 10 和 Windows 11 的蓝牙设备电量管理工具，目标是解决系统原生蓝牙电量展示入口深、信息不集中、提醒不及时的问题。

项目计划使用 C# 实现桌面端应用，提供后台蓝牙设备扫描、设备连接状态监控、电量读取、系统托盘展示、任务栏/贴边电量展示、低电量提醒、异常日志记录和短时弹窗提示能力。

## 2. 目标平台与技术选型

| 项目 | 说明 | 备注 |
| --- | --- | --- |
| 目标系统 | Windows 10、Windows 11 | 优先适配 Windows 10 1903 及以上版本、Windows 11 当前主流版本 |
| 开发语言 | C# | 便于调用 Windows 蓝牙、通知、托盘、窗口相关 API |
| UI 框架 | WPF | 兼容 Windows 10/11，部署成本低，适合桌面工具类应用 |
| 运行时 | .NET 8 或 .NET 6 | 优先 .NET 8；如需更保守兼容，可降级到 .NET 6 LTS |
| 蓝牙能力 | Windows.Devices.Bluetooth、Windows.Devices.Enumeration、Windows Runtime API、必要时补充 WMI/注册表读取 | BLE 与 BTC 的电量获取通道不同，需要分层适配 |
| 托盘能力 | NotifyIcon / H.NotifyIcon.Wpf 或 WinForms NotifyIcon | WPF 原生没有完整托盘组件，可引入轻量封装 |
| 任务栏/贴边展示 | 托盘图标 + 可选桌面边缘悬浮窗 | Windows 任务栏不建议直接注入自定义多图标区域，需规避兼容和遮挡风险 |
| 日志 | Microsoft.Extensions.Logging + Serilog | 支持本地滚动日志、异常上下文记录 |
| 配置存储 | JSON 配置文件 | 存储设备展示开关、刷新间隔、低电量阈值等 |

## 3. 总体架构

项目采用分层架构，核心业务与 UI 展示分离，便于后续替换蓝牙读取实现或增加更多展示方式。

| 层级 | 模块 | 职责 |
| --- | --- | --- |
| 应用入口层 | App Bootstrapper | 初始化日志、配置、依赖注入、主窗口、托盘服务、后台扫描服务 |
| UI 展示层 | Main Window、Device List、Settings、Toast Window、Taskbar Overlay | 展示设备列表、电量、连接状态、错误提示和用户设置 |
| 业务服务层 | Device Monitor、Battery Aggregator、Alert Service、Display Preference Service | 周期扫描、状态合并、低电量判断、展示策略管理 |
| 蓝牙适配层 | BLE Adapter、Classic Bluetooth Adapter、Device Enumeration Adapter | 获取已配对/已连接设备、读取电量、解析连接状态 |
| 基础设施层 | Logging、Configuration、Persistence、Notification、Scheduler | 日志、配置读写、本地持久化、系统通知和定时任务 |

## 4. 核心数据模型

| 模型 | 字段 | 说明 |
| --- | --- | --- |
| BluetoothDeviceInfo | DeviceId、Name、Address、DeviceType、IsConnected、BatteryLevel、LastSeenAt、LastUpdatedAt | 应用内部统一设备状态模型 |
| DeviceType | BLE、ClassicBluetooth、Unknown | 标记设备类型，便于选择不同电量读取策略 |
| DeviceDisplayPreference | DeviceId、ShowInTray、ShowInTaskbarOverlay、Alias、SortOrder | 保存每个设备的展示偏好 |
| BatteryReadResult | Success、BatteryLevel、ErrorCode、ErrorMessage、Source | 表示一次电量读取结果 |
| AppSettings | ScanIntervalSeconds、LowBatteryThreshold、EnableOverlay、EnableErrorPopup、LogLevel | 全局配置 |
| ErrorEvent | Time、Module、DeviceId、Operation、Exception、UserMessage | 记录可追踪的异常事件 |

## 7. 开发与调试确认规则

后续项目实现必须遵循“一次只交付一个可验证功能”的节奏。每完成一个功能后，需要先进入用户调试确认阶段，用户确认无误后，才继续实现下一个功能。

| 规则 | 说明 | 状态 | 备注 |
| --- | --- | --- | --- |
| 单功能交付 | 每次只完成一个明确功能点或一组强相关的小功能 | 已完成 | 避免多个功能混在一起导致问题定位困难 |
| 调试方式说明 | 每完成一个功能后，必须给出本地运行、操作路径、预期结果和异常排查方式 | 已完成 | 如功能依赖 Windows 蓝牙硬件或系统权限，需要明确说明测试前置条件 |
| 用户确认门禁 | 用户调试确认无误后，再继续实现下一个功能 | 已完成 | 若用户反馈问题，优先修复当前功能，不跳到新功能 |
| 文档同步 | 功能完成、部分完成、失败、遗留问题都必须同步写入本文档 | 已完成 | 对应功能表需要更新进度和备注 |

| 调试证据 | 能自动验证的功能尽量提供命令、测试用例或日志位置 | 已完成 | UI/托盘/蓝牙硬件能力无法完全自动验证时，提供手工调试清单 |

## 8. 任务栏展示风险与处理方案

用户需求中提到“无需打开 UI 界面，即可在底部任务栏中查看设备电量”，同时要求“当系统托盘或任务栏中有新打开的其他应用，电量图标能够自适应改变位置，以防图标重叠和穿模”。

在 Windows 桌面应用中，普通应用直接向系统任务栏内部插入多个自定义动态图标并精确避让其他应用图标，不属于稳定、推荐的公开能力，存在系统版本差异、DPI 缩放异常、安全限制和系统更新后失效的风险。

因此本项目建议采用以下实现策略：

| 方案 | 说明 | 进度 | 备注 |
| --- | --- | --- | --- |
| 托盘摘要图标 | 使用系统通知区域展示应用状态和最低电量摘要 | 未完成 | 稳定性最高，兼容 Windows 10/11 |
| 贴边电量条 | 使用置顶、无边框、可点击穿透控制的 WPF 悬浮窗贴近任务栏上方或侧边 | 未完成 | 可实现“底部可见电量”，并规避任务栏内部注入风险 |
| 自适应布局 | 根据屏幕工作区自动调整贴边窗口位置 | 未完成 | 不直接依赖任务栏内部图标坐标 |
| 空间不足降级 | 设备过多时显示用户勾选设备、最低电量摘要或横向滚动 | 未完成 | 避免遮挡和穿模 |
| 用户开关 | 允许用户关闭贴边电量条，只保留托盘 | 未完成 | 降低对全屏应用或特殊桌面环境的影响 |
